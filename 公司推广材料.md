# VibeSOP
## 金融科技企业 AI 辅助开发解决方案

> 经过实战检验的 AI 编程工作流基础设施——让 AI 辅助开发真正落地生产

---

## 一、背景与问题

### 当前 AI 辅助开发的痛点

| 痛点 | 具体表现 | 金融行业影响 |
|------|----------|-------------|
| **无记忆** | AI 每次会话从零开始,不记得上次的决策和教训 | 重复踩坑,技术债务积累 |
| **无验证** | AI 声称"完成了"但代码未经测试 | 线上故障风险,合规隐患 |
| **无结构** | 每个人用 AI 的方式不同,质量参差不齐 | 团队协作困难,代码审查成本高 |
| **无追溯** | 开发决策散落在聊天记录中,无法审计 | 监管合规风险 |
| **成本失控** | 高能力模型用于所有任务,API 费用高 | ROI 难以量化 |

---

## 二、解决方案概述

VibeSOP 是一套**结构化工作流配置框架**,为 Claude Code、OpenCode 等 AI 编程工具提供:

```
┌─────────────────────────────────────────────────────┐
│                  结构化规则层 (rules/)                 │  ← 始终加载,核心行为约束
├─────────────────────────────────────────────────────┤
│                  参考文档层 (docs/)                    │  ← 按需加载,最佳实践指南
├─────────────────────────────────────────────────────┤
│                  工作记忆层 (memory/)                  │  ← 热数据,项目状态和知识
└─────────────────────────────────────────────────────┘
```

**核心理念**:
- 结构化 > 临时提示词
- 记忆积累 > 每次重新开始
- 验证通过 > 主观自信
- 可追溯 > 黑盒操作

## 三、核心价值（管理层视角）

### 3.1 降低合规风险

金融科技企业面临严格的监管要求,AI 辅助开发必须满足:

- **决策可追溯**: 所有架构决策、技术选型自动记录到 `memory/project-knowledge.md`
- **变更可审计**: 结构化的会话记录,每次工作进度有据可查
- **单一事实来源 (SSOT)**: 防止信息散落导致的不一致性,满足审计要求
- **安全审查内置**: 代码安全审查 Agent 内置 OWASP Top 10 检查清单

### 3.2 提升研发效率

| 指标 | 传统方式 | 使用本框架 | 提升 |
|------|----------|-----------|------|
| 新人上手时间 | 2-4 周 | 3-5 天 | ~70% |
| 重复 Bug 率 | 高(无记忆) | 低(经验积累) | 显著降低 |
| 代码审查时间 | 人工全量审查 | AI 预审 + 人工复核 | ~40% |
| API 调用成本 | 全部用高端模型 | 按任务复杂度路由 | 30-60% |

### 3.3 标准化团队实践

- 统一的 AI 使用规范,消除"各自为政"
- 可复用的技能库(TDD、代码审查、系统调试)
- 项目级配置覆盖(overlay),支持不同业务线定制

## 四、技术特性（研发团队视角）

### 4.1 智能任务路由

根据任务复杂度自动匹配合适的模型,避免资源浪费:

```
critical_reasoner  → Claude Opus   (安全审查、架构决策、关键业务逻辑)
workhorse_coder    → Claude Sonnet (日常开发、代码重构、功能实现)
fast_router        → Claude Haiku  (快速查询、简单任务、子流程)
independent_verifier → 第二模型    (交叉验证、代码审查)
cheap_local        → 本地模型      (提交信息、格式化)
```

**金融场景示例**:
- 支付核心逻辑修改 → `critical_reasoner` (Opus)
- 日常 CRUD 接口开发 → `workhorse_coder` (Sonnet)
- 查询 API 文档 → `fast_router` (Haiku)

### 4.2 内置安全审查

专用安全审查 Agent,自动检查:
- SQL 注入、XSS、CSRF 等 OWASP Top 10 漏洞
- 敏感数据泄露(PII、密钥、证书)
- 认证授权问题
- 金融业务特有的安全风险

### 4.3 强制验证机制

**"验证优于自信"原则**: AI 在声称任务完成前,必须:
1. 运行测试命令并读取输出
2. 确认测试通过,而非假设通过
3. 提供验证证据

这直接消除了 AI 编码最常见的失败模式。

### 4.4 结构化调试流程

五阶段系统调试技能:
1. **召回** - 检查历史记录中的类似问题
2. **根因分析** - 深入调查,不做表面修复
3. **修复** - 针对根因的精准修复
4. **验证** - 确认修复有效
5. **记录** - 将经验写入知识库

## 五、金融科技场景应用

### 场景 1: 核心交易系统开发

```yaml
# 项目级配置 (.vibe/overlay.yaml)
profile:
  mapping_overrides:
    critical_reasoner: claude.opus-class  # 交易逻辑必须用最高能力模型

policies:
  append:
    - id: financial-compliance
      summary: "所有涉及资金流转的代码变更必须经过安全审查"
    - id: audit-trail
      summary: "记录所有架构决策到 project-knowledge.md"
```

### 场景 2: 风控模型开发

- 使用 `systematic-debugging` 技能追踪模型异常
- 使用 `verification-before-completion` 确保模型测试通过
- 使用 `experience-evolution` 积累风控规则知识库

### 场景 3: 监管报送系统

- 结构化记忆确保报送逻辑变更可追溯
- 安全审查 Agent 检查数据脱敏是否到位
- 会话记录满足变更管理要求

## 六、实施方案

### 阶段一: 试点 (2-4 周)

1. 选择 1-2 个非核心项目试点
2. 安装配置框架
3. 培训开发团队核心概念
4. 收集反馈,调整配置

### 阶段二: 推广 (1-2 个月)

1. 制定公司级 overlay 配置(统一安全策略、合规要求)
2. 推广到更多项目团队
3. 建立内部知识库和最佳实践

### 阶段三: 深化 (持续)

1. 开发金融业务专用技能(风控审查、合规检查)
2. 与内部 CI/CD 流程集成
3. 建立 AI 辅助开发效能度量体系

## 七、快速开始

```bash
# 1. 克隆框架
git clone https://github.com/nehcuh/vibesop.git
cd vibesop

# 2. 安装 vibe 命令
bin/vibe-install

# 3. 初始化 Claude Code 配置
vibe init --platform claude-code

# 4. 在你的项目中应用
cd /path/to/your-fintech-project
vibe switch --platform claude-code

# 5. 启动 AI 辅助开发
claude
```
